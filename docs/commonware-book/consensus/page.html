Every replicated system eventually has to answer one simple and uncomfortable question: when several machines disagree about what should happen next, who gets to decide? In a Byzantine environment — where participants can lie, equivocate, or disappear on purpose — the answer has to be a protocol, not a convention.

commonware-consensus provides that protocol surface for Commonware systems. It orders opaque messages in a Byzantine environment. Opaque is the load-bearing word: consensus never needs to parse the application payload. It only orders digests. The application decides what deserves a digest; consensus decides how those digests enter history.

The crate is organized as four layered primitives:

  • simplex (BETA): A three-phase Byzantine consensus protocol with leader rotation and threshold signature support.
  • aggregation (ALPHA): Recoverable quorum certificates over an externally synchronized sequence of items.
  • ordered_broadcast (ALPHA): Reliable, ordered delivery from sequencers to validators with chaining and journaling.
  • marshal (ALPHA): Coordinates broadcast delivery with simplex finalization, handles backfill, and delivers finalized blocks to the application.
Source note

See consensus/src/lib.rs for the trait hierarchy; consensus/src/simplex/actors/voter/actor.rs for the main voter state machine; consensus/src/marshal/mod.rs for the marshal actor.

Backgrounder: Why Consensus Exists At All

Consensus sounds formal very quickly. Agreement. Liveness. Byzantine faults. Quorums. Those words are important, but the broad question underneath them is simpler: once more than one machine is allowed to matter, who gets to decide what the shared history is?

If one machine owns the state, the next action is whatever that machine says it is. Replication changes the problem. Several nodes now hold related state, and the system wants availability, performance, or trust properties that no single node should control alone. So the real task becomes keeping many local views from drifting into several incompatible histories.

From State Machines to Shared History

The clean undergraduate-level picture is the replicated state machine. If all honest nodes start from the same state and apply the same commands in the same order, they stay in sync. So consensus is usually not about understanding the full application payload. It is about deciding which command, block, or digest occupies the next slot in history.

That is why Commonware consensus orders opaque digests. The application decides what those digests mean. Consensus decides how one shared sequence emerges from competing proposals under adversarial conditions.

Why Voting Alone Is Not Enough

A beginner's first idea is often “let everyone vote and take the majority.” That helps in one narrow instant, but it does not solve the whole problem. A majority observed by whom, at what time, over which proposal, and with what guarantee that a different observer will not see a different majority built from delayed messages?

Once the network can delay messages or a malicious leader can equivocate, “count the votes” is not a protocol. The system needs enough support for one history that any conflicting support must overlap with it. That is the idea behind quorums: geometry under a fault assumption.

Safety, Liveness, and Why Timeouts Exist

Consensus protocols live between two big obligations. Safety says honest nodes must not finalize conflicting histories. Liveness says the protocol must keep moving once the network becomes well behaved enough. These goals pull in different directions. A protocol can always preserve safety by waiting longer, but then progress stalls. It can push for progress aggressively, but then the risk of conflicting or unsafe movement rises.

This is why leaders, rounds, and timeouts are not implementation details. They are the practical answer to a deeper uncertainty: the system cannot always tell whether a leader is slow, dead, or malicious. So it gives the leader a turn, waits long enough to gather evidence, and then moves to a new view when that evidence does not arrive.

Why Byzantine Consensus Needs Evidence

Crash-fault tolerance already needs coordination. Byzantine fault tolerance is harsher because faulty participants may lie, equivocate, replay, or strategically omit messages. Once that is allowed, signatures and certificates become load-bearing. The protocol needs more than "someone said yes." It needs durable evidence that enough support accumulated under rules that make conflicting histories unsafe.

That is why certificates are more than receipts. They are state-carrying objects that constrain what future rounds are allowed to do.

Background Takeaway

Consensus exists because replicated systems need one shared history in a world where messages are delayed and some participants may behave strategically. The protocol's real job is to turn local votes into global evidence without sacrificing either safety or eventual progress.

1. What Problem Does This Solve?

Distributed consensus asks a deceptively small question: given a network of N processes where up to f may behave arbitrarily, how do the honest ones agree on the same sequence of decisions?

The classical answer is PBFT and its descendants. Those protocols work, but the happy path is expensive: many message phases, a lot of all-to-all traffic, and enough ceremony per decision that large validator sets feel the weight of it.

commonware-consensus takes a different route. Simplex keeps the common case to three phases — notarize, nullify, finalize — and pushes as much work as possible into certificates that can be reused across rounds. A certificate is not just a receipt for the past. It is the parent context for what comes next.

The design is built around three properties:

  1. Safety: No two honest nodes finalize different values at the same height. This holds as long as fewer than N/3 participants are Byzantine.
  2. Liveness: The system continues to make progress under partial synchrony, given an eventually leader. Timeout and view-change mechanisms ensure the protocol does not stall indefinitely.
  3. Attributability: With attributable schemes (ed25519, bls12381_multisig, secp256r1), the signature on a notarize or finalize vote can be used as evidence of a fault. With threshold schemes (bls12381_threshold), signatures cannot be attributed to individual validators — useful for internal liveness proofs but not for punishment.

The crate also separates concerns that many consensus implementations tangle together:

  • What is being decided (the payload) is opaque to consensus. The Automaton trait provides digests; consensus orders them.
  • Who decides (leader election) is pluggable. RoundRobin gives deterministic rotation; Random gives unpredictable leaders via threshold VRF.
  • How signatures are aggregated is pluggable. Attributable schemes preserve fault evidence; threshold schemes give succinct certificates.

2. Mental Model

Think of simplex as a voting machine with memory. Each round, called a view, has a designated leader. The leader proposes a block. Validators then move through three kinds of votes:

Propose → Notarize → Nullify → Finalize
                     ↑
                     …  If leader fails or block invalid: timeout → view change

Notarize asks: is this proposal valid enough to build on? A validator who accepts the proposal signs a notarize vote. Once more than 2N/3 validators do that, the block has a notarization certificate. The block is not yet finalized, but it has crossed an important line: any safe future must account for it.

Nullify asks: should we abandon this view and move on? If the leader stalls or proposes nonsense, validators can vote to skip the view. But nullify votes are not free-floating. They still have to respect parent continuity. You cannot safely skip view v unless you can also prove you are not jumping over certified history that honest validators may already depend on.

Finalize asks: are we ready to commit this block as part of the durable sequence? A block can only be finalized after it has already been notarized. That ordering is the conceptual center of the protocol: first establish that the proposal is admissible, then establish that the system is ready to commit it.

Invariant

Finalization requires notarization, and notarization requires proposal validity. Once that chain is in your head, the rest of the protocol is much easier to reason about.

Composing the Layers

The four primitives stack like this:

marshal          ← delivers finalized blocks to the application
    ↑              (coordinates broadcast + simplex + backfill)
simplex          ← three-phase consensus
    ↑              (coordinates voter + batcher + resolver)
aggregation      ← quorum certificates over external sequence
    ↑              (coordinates with externally-provided ordered items)
ordered_broadcast← sequencer → validator reliable broadcast

Each layer solves a different failure problem. simplex handles Byzantine leaders and view changes. ordered_broadcast makes sequencer equivocation detectable through chaining. aggregation turns externally provided ordered items into quorum-certified facts. marshal closes the gap between "a block exists somewhere" and "the application can now consume finalized blocks in order."

3. Core Abstractions and Important Types

This chapter gets easier if you sort the types by the question they answer: What time are we talking about? What is being voted on? Who gets to influence the decision? How does data reach validators? In that order, the type list reads like a model of the protocol instead of a glossary dump.

Identifier Types (consensus/src/types.rs)

The protocol uses four different notions of "where we are." Keeping them separate prevents a lot of confusion later.

Epoch — the validator-set boundary. If membership changes, the epoch changes. Certificates do not cross that boundary.

View — the leader-attempt counter inside one epoch. If a leader fails, the view increments.

Round — the pair (Epoch, View). This is the protocol's full "where are we right now?" coordinate.

Height — the position of a finalized block in the output chain. Heights are application-visible history. Rounds are protocol progress. One height may need several rounds before it is finalized.

Arithmetic

All identifier arithmetic is overflow-safe. next() panics on overflow (which is appropriate for identifiers that should never realistically overflow). Arithmetic that might legitimately underflow returns Option or saturates.

The Trait Hierarchy (consensus/src/lib.rs)

Automaton is the boundary between consensus and the application. It tells consensus three things: what genesis digest to start from, how to ask the application for a proposal, and how to ask whether that proposal is acceptable.

pub trait Automaton: Clone + Send + 'static {
    type Context;
    type Digest: Digest;

    fn genesis(&mut self, epoch: Epoch) -> impl Future<Output = Self::Digest> + Send;
    fn propose(&mut self, context: Self::Context) -> impl Future<Output = oneshot::Receiver<Self::Digest>> + Send;
    fn verify(&mut self, context: Self::Context, payload: Self::Digest) -> impl Future<Output = oneshot::Receiver<bool>> + Send;
}

The important abstraction choice is that consensus traffic carries digests, not full application objects. That keeps consensus generic and keeps large payload movement in the broadcast layer where it belongs.

propose and verify return one-shot channels rather than plain booleans because the application may need time to decide. But once the channel resolves or closes, consensus treats the verdict as final for that attempt.

Single-shot semantics

Once the returned channel resolves or closes, consensus treats verification as concluded and will not retry. If the application is still uncertain, it must keep the channel open. Returning false is permanent — consensus will never retry. The application must only return false when the payload is permanently invalid.

CertifiableAutomaton extends Automaton with a certify phase between notarization and finalization. This is where the application can say: "The block is valid, but I am not yet ready to finalize it."

pub trait CertifiableAutomaton: Automaton {
    fn certify(&mut self, round: Round, payload: Self::Digest)
        -> impl Future<Output = oneshot::Receiver<bool>> + Send {
        async move { receiver }
    }
}

The certify decision must be deterministic across honest participants. If equally honest validators make different certify decisions on the same input, liveness falls apart.

Relay broadcasts full payloads to the network. Consensus only knows digests; the relay disseminates the data.

Reporter emits activity events (votes, finalizations, faults) for reward/penalty systems.

Monitor allows external actors to subscribe to consensus progress (latest finalized index).

Simplex Types (consensus/src/simplex/types.rs)

Subject<'a, D> identifies what a validator is actually signing at a given moment:

pub enum Subject<'a, D: Digest> {
    Notarize { proposal: &'a Proposal<D> },
    Nullify  { round: Round },
    Finalize { proposal: &'a Proposal<D> },
}

That split is worth noticing. Notarize and Finalize both point at a proposal. Nullify points only at a round, because a nullify vote is not endorsing data; it is endorsing progress to the next attempt.

VoteTracker<S, D> accumulates votes per phase:

pub struct VoteTracker<S: Scheme, D: Digest> {
    notarizes: AttributableMap<Notarize<S, D>>,
    nullifies: AttributableMap<Nullify<S>>,
    finalizes: AttributableMap<Finalize<S, D>>,
}

AttributableMap<T> is keyed by validator index, so each validator gets at most one slot per phase. That is how the code enforces the familiar rule "one validator, one vote, per phase, per round."

Context<D, P> carries round metadata for proposal verification:

pub struct Context<D: Digest, P: PublicKey> {
    pub round:    Round,
    pub leader:   P,
    pub parent:   (View, D),
}

The parent field is the critical piece. If there is a gap between the current view and the parent view, the validator must possess a nullification for each discarded view before safely voting on a new proposal. This is the fork-safety mechanism.

Signing Schemes (consensus/src/simplex/scheme/mod.rs)

The scheme layer is where attribution policy is decided. All schemes implement certificate::Scheme<Subject>:

  • ed25519, bls12381_multisig, secp256r1: Attributable. Each individual signature carries the signer’s identity. Certificates contain (index, signature) pairs.
  • bls12381_threshold: Non-attributable. Any t partial signatures can be combined to forge a signature for any other participant. Certificates contain only the recovered threshold signature.
Non-attribution implication

With bls12381_threshold, evidence cannot be exported to third parties. A signature that proves validator V participated in a quorum cannot be verified by an external observer — because an external observer cannot distinguish a genuine V signature from a forged one. For punishment protocols, use attributable schemes.

Domain separation is enforced via namespace suffixes:

const NOTARIZE_SUFFIX: &[u8] = b"_NOTARIZE";
const NULLIFY_SUFFIX:  &[u8] = b"_NULLIFY";
const FINALIZE_SUFFIX: &[u8] = b"_FINALIZE";

A signature made under _NOTARIZE cannot be replayed as a _FINALIZE signature. This prevents cross-protocol attacks where a notarization certificate could be repurposed as a finalization certificate.

Ordered Broadcast (consensus/src/ordered_broadcast/)

The system has two roles: sequencers broadcast chunks; validators sign chunks to form quorum certificates.

Node is the fundamental broadcast unit:

pub struct Node<P: PublicKey, D: Digest> {
    pub chunk:  Chunk<P, D>,
    pub parent: Option<Certificate<P>>,
}

Each node contains a chunk and a certificate over the previous chunk from the same sequencer. This creates a chain: to verify node at height h, you need the certificate for height h-1. A sequencer that broadcasts two different chunks at the same height produces a fork — detectable and attributable.

Aggregation (consensus/src/aggregation/)

Aggregation differs from ordered_broadcast in a subtle way: the sequence of items is externally provided (by some other consensus process), and validators only need to certify that they agree on the sequence.

Item<D> is an externally-provided (height, digest) pair. The aggregation engine monitors an external source of Items, collects acknowledgments from validators, and assembles certificates. Unlike ordered_broadcast, there is no chaining: each item is certified independently based on its height and digest.

Marshal (consensus/src/marshal/)

Marshal bridges consensus and the application. It:

  1. Receives uncertified blocks from a broadcast mechanism (e.g., commonware_broadcast::buffered).
  2. Receives notarizations and finalizations from simplex.
  3. Reconstructs total order via core::Actor.
  4. Provides backfill when gaps appear.

The actor delivers blocks at-least-once, in monotonically increasing order. The application must handle duplicates. Finalization is view-based: a block is finalized when marshal has seen a valid finalization certificate from consensus for that block.

Starting height (floor): Marshal can be configured to only store blocks above a certain height. This enables state sync from a recent snapshot rather than from genesis. Setting this height prevents peers from backfilling blocks below it.

4. Execution Flow and Lifecycle

This is the point where the protocol stops being a list of types and becomes a moving system. The main thing to watch is not the mailbox plumbing by itself. It is how the actor flow preserves the earlier invariants while messages arrive late, certificates appear out of order, and timeouts fire.

Simplex Voter Lifecycle

The voter actor (consensus/src/simplex/actors/voter/actor.rs) is the heart of simplex. It runs a select! loop handling three classes of events:

  1. Incoming messages (proposals, votes, certificates) via the mailbox.
  2. Timeout events from the deterministic clock.
  3. Automaton responses (verification, certification verdicts).

The loop uses select_loop! from commonware_macros which is a tokio-aware select that handles cancellation correctly.

On receiving a proposal:

  1. Extract context (round, leader, parent) from proposal.
  2. Check if we already have a notarize/finalize for this view → ignore if duplicate.
  3. Check parent nullification: if parent.view < our latest known view, ensure we have nullifications for all views in between. If not → store proposal but don’t vote yet.
  4. Send verification request to automaton.
  5. On verification true → send notarize vote to batcher. On false → stay silent.

On receiving a notarization certificate (>2N/3 notarizes):

  1. Record the notarization certificate in state.
  2. Send certification request to automaton (via CertifiableAutomaton).
  3. On certification true → send finalize vote to batcher. On false → don’t finalize.

On timeout (no progress in current view):

  1. Send nullify vote to batcher.
  2. If >2N/3 nullifications accumulated → enter view change.
  3. In view change: advance view, reset vote trackers, request certificates from resolver.

Marshal Block Ordering

Marshal coordinates three input streams:

  1. Uncertified blocks from broadcast (via commonware_broadcast::buffered or coding::shards::Engine).
  2. Notarizations from simplex.
  3. Finalizations from simplex.

It tracks a view-based state machine:

For each view v:
  - Wait for block at height h with notarization cert
  - If block's parent is finalized and we have cert for v:
      - Finalize block and all ancestors
      - Deliver to reporter
  - If timeout:
      - Backfill request for missing blocks

Backfill: When marshal detects a gap (it knows a finalization for height h but doesn’t have the block at h-1), it queries the resolver for the missing block. Resolver queries peers. Once the gap is filled, finalization proceeds.

Standard vs. coding mode: In standard mode, marshal receives blocks via commonware_broadcast::buffered. In coding mode, it receives erasure-coded shards from coding::shards::Engine and reconstructs the full block before finalizing.

5. Concurrency, Protocol, and Systems Semantics

These are the disciplines the code is trying to enforce. A consensus protocol is not only a sequence of messages; it is a set of promises about quorum overlap, evidence, determinism, and recovery. The implementation details matter because this is where those promises either survive contact with the runtime or quietly disappear.

Deterministic Runtime and Testing

All consensus actors run on commonware_runtime::deterministic::Runner. This is a simulated runtime that controls time explicitly, enabling deterministic test execution. Tests can advance time manually, fire multiple concurrent events, and verify deterministic outcomes.

The deterministic runtime is why the codebase can test Byzantine scenarios reproducibly: with a fixed seed, the same sequence of events always produces the same result. This is critical for testing protocols where timing matters (timeouts, view changes, competing proposals).

In practice the tests make that reproducibility inspectable. Validators, reporters, applications, engines, and the network are labeled with with_label(...); many scenarios return context.auditor().state() as a compact witness of the schedule that actually ran; restart paths use start_and_recover(); and adversarial cases inspect oracle.blocked() to confirm that peers the protocol meant to quarantine were actually blocked.

Actor Message Passing

All simplex actors — voter, batcher, resolver — communicate via typed channels. There is no shared mutable state between actors. The voter sends vote artifacts to the batcher, which batches and broadcasts them. The resolver receives certificates from the network and provides them to the voter. This isolation means the protocol is testable in isolation: you can test the voter state machine with mock batcher and resolver.

N3f1 Quorum Threshold

Throughout the crate, quorum is N3f1::quorum(N) = N - N/3 (ceiling). For N=4, quorum is 3; for N=7, quorum is 5; for N=10, quorum is 7. This is the minimum quorum that guarantees:

  • Safety: With f = N/3 Byzantine nodes, any two quorums of size >2N/3 must overlap in at least one honest node.
  • Liveness: An honest quorum can always make progress (since N - f = 2f + 1 > f honest nodes).

Journaling and Crash Recovery

Both ordered_broadcast and simplex use journaling for crash recovery. Events (votes, certificates, node broadcasts) are written to a journal before being acted on. On restart, the journal is replayed: the actor re-constructs state from the persisted log.

For ordered_broadcast, the journal prevents sequencer equivocation: if a sequencer crashes after broadcasting node at height h but before persisting, on restart it will find the journal entry and know not to broadcast a different chunk at height h.

For simplex, the journal stores vote artifacts and certificates. On restart, the voter re-syncs with peers via the resolver to fill gaps.

VRF Randomness and the Commit-Then-Reveal Pattern

Security warning

It is not safe to use a round’s randomness to drive execution in that same round. A malicious leader can selectively distribute blocks to gain early visibility of the randomness output, then choose nullification if the outcome is unfavorable. The recommended pattern is commit-then-reveal: bind randomness requests in finalized blocks at view v for use at view v+100 (or any sufficiently distant view).

6. Failure Modes, Cancellation, and Correctness Concerns

This section is the protocol under stress. The happy path tells you what the design wants to do. The failure sections tell you what it is actually prepared to survive, what it can only mitigate, and where the hard limits really are.

Byzantine Fault Tolerance Limits

The N/3 tolerance is a hard limit. If more than N/3 participants are simultaneously Byzantine:

  • Safety can be violated: A Byzantine quorum can finalize two conflicting blocks at the same height (by controlling which blocks propagate to which honest nodes).
  • Liveness can be violated: A Byzantine coalition can always have a quorum to block finalization.

In practice, this means the system is secure against up to 33% adversarial participation. For 10 validators, 3 can be Byzantine. For 100 validators, 33 can be Byzantine.

Cancellation and Timeouts

The voter uses leader_timeout and certification_timeout to bound how long it waits. The invariant enforced at construction:

if cfg.leader_timeout > cfg.certification_timeout {
    panic!("leader timeout must be less than or equal to certification timeout");
}

The leader timeout must not exceed the certification timeout because certification (the application’s certify call) happens after notarization. If the leader timeout were larger, a leader could propose, get notarized, then timeout before certification completes — causing unnecessary view changes.

Marshal Gap Handling

Marshal delivers blocks in monotonically increasing order (no gaps). When a gap appears:

  1. Marshal detects that it has a finalization for height h but no block at h-1.
  2. It pauses finalization of h and emits a backfill request.
  3. Resolver queries peers for the missing block.
  4. On receipt, Marshal verifies and delivers h-1.
  5. Finalization of h resumes.

If the gap cannot be filled (peer doesn’t have it, or peer is malicious), Marshal will continue to retry. The application receives no updates until the gap is resolved.

Double-Signing and Equivocation Detection

In ordered_broadcast, a sequencer that broadcasts two different chunks at the same height with the same parent produces detectable equivocation: two valid signatures from the same validator on the same height with different payloads. The ordered_broadcast::types::Error::ChunkMismatch error captures this.

In simplex, a validator that sends two different notarize votes at the same view produces equivocation. The AttributableMap enforces one vote per phase per validator: the second insert returns false.

7. How to Read the Source Code

Read the consensus crate as an argument, not as a catalog. Start with the temporal identifiers and the trait boundary to the application. Then read the vote subjects and certificate machinery. Only after that should you move outward into the voter state machine and the layers built above it.

Reading Order

For understanding the protocol:

  1. consensus/src/types.rs — understand the identifier types (Epoch, View, Height, Round). All consensus decisions are scoped to these identifiers.
  2. consensus/src/lib.rs — understand the trait hierarchy. Automaton is the key interface.
  3. consensus/src/simplex/types.rs — understand Subject, VoteTracker, and Context. These encode the core protocol state.
  4. consensus/src/simplex/scheme/mod.rs — understand domain separation (namespace suffixes). This is where cross-protocol attacks are prevented.
  5. consensus/src/simplex/elector.rs — understand leader election determinism requirements.
  6. consensus/src/simplex/actors/voter/actor.rs — the main state machine. Read this last and carefully.

For understanding the layers above simplex:

  1. consensus/src/ordered_broadcast/types.rs — Node chaining, certificate assembly.
  2. consensus/src/aggregation/types.rs — Item/Ack/Certificate model.
  3. consensus/src/marshal/mod.rs and marshal/core.rs — how blocks flow from broadcast through consensus to the application.

Key Invariants to Verify in Code

  • Parent nullification: In voter/actor.rs, search for where parent context is checked before sending notarize votes.
  • Single-shot verification: In lib.rs, the doc comment on verify explains the single-shot property. In actor.rs, look for how the oneshot channel is used.
  • Quorum assembly: In scheme/*/mod.rs, look for where N3f1::quorum is used to determine when a certificate is complete.
  • Domain separation: In scheme/mod.rs, look for the suffix constants and how they are used in Namespace::derive.
  • Deterministic election: In elector.rs, the Elector::elect trait method has a determinism requirement in its docs.

8. Glossary

Attributable signature
A signature that uniquely identifies the signer. With an attributable scheme, any party can verify that a specific validator signed a specific message.
Certificate
A quorum of signatures (or a threshold signature) over a common subject, proving that >2N/3 validators have voted on the same value.
Domain separation
The technique of deriving distinct namespaces for different purposes so that a signature/primitive created for one purpose cannot be used for another. In simplex, _NOTARIZE, _NULLIFY, _FINALIZE provide domain separation.
Epoch
A period of time during which the validator set is fixed. Epochs increment on reconfiguration.
Finalization
The point at which a block is considered committed. After finalization, the block cannot be reverted without a coordinated chain reorg.
Leader election
The process of selecting which validator proposes the next block. In RoundRobin, this is deterministic. In Random, this uses VRF-derived randomness.
N3f1 quorum
The quorum size N - N/3 (ceiling), which ensures overlap with any other quorum of the same size in a network with N/3 Byzantine participants.
Notarization
A vote by a validator that a proposal is valid. A notarization certificate proves that >2N/3 validators have verified and accepted the proposal.
Nullification
A vote to skip a view. Nullification certificates allow the next leader to safely skip a view without risking a fork.
Simplex
The three-phase (notarize → nullify → finalize) consensus protocol in commonware-consensus.
Threshold signature
A signature produced by combining partial signatures from multiple participants. The combined signature does not reveal which individual participants signed.
View
A single consensus round within an epoch, identified by (epoch, view_number). Each view has a designated leader.
VRF (Verifiable Random Function)
A function that produces a pseudorandom output that can be verified by anyone given a public key. In Random leader election, VRF outputs determine the leader.

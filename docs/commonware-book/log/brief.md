# commonware-log - Interactive Book Chapter Brief

## 1. Module Purpose

`commonware-log` is a lecture in boundary setting. It shows how to keep a
payload private while still letting consensus agree on a public commitment to
it. The example is not a full log service. It is a focused lesson in
separation of concerns: the application keeps the raw message local, hashes it,
and lets consensus agree on the digest as the public fact.

That distinction is the whole point. A naive design would push the whole
payload through the agreement layer, which would leak secret data, inflate the
protocol surface, and make the consensus layer care about payload handling it
should never own. `commonware-log` keeps the secret private, commits to it with
a digest, and persists consensus state so the story continues after restart.

The example also makes one deliberate omission obvious: it does not try to
broadcast raw messages or backfill missing payloads. The book chapter should use
that omission as part of the lesson. Consensus is being used to agree on a
commitment, not to distribute the commitment's contents.

## 2. Key Source Files

### `examples/log/src/main.rs`
Wires the example together. It parses identities, bootstraps the network,
creates the application and consensus engine, and starts the terminal UI. This
file is the best place to see the shape of the whole system without yet getting
lost in details.

### `examples/log/src/application/mod.rs`
Defines the application boundary. The `Config` type shows the minimum the
example needs: a hasher, a signing scheme, and a mailbox size. The `Scheme`
alias makes clear that this example uses the consensus scheme directly.

### `examples/log/src/application/actor.rs`
The heart of the example. This actor generates a 16-byte secret message on its
turn, hashes it, and sends the digest back to consensus. It also handles the
genesis digest and logs the secret locally for observability.

### `examples/log/src/application/ingress.rs`
Implements the consensus-facing mailbox. This is where the application becomes
an `Automaton`, `CertifiableAutomaton`, and `Relay`. The file shows the exact
boundary where consensus asks for digests and the application decides what to
do with them.

### `examples/log/src/application/reporter.rs`
Turns consensus activity into structured logs. This is the observer's window
into the protocol: notarized, finalized, and nullified payloads.

### `examples/log/src/gui.rs`
Purely a reading aid. It formats progress and log output into a TUI so the
example is easier to follow, but it does not change the protocol story.

## 3. Chapter Outline

1. The real problem: deciding what stays private and what becomes public.
2. The mental model: a locked envelope with a public receipt.
3. Why the digest is the thing that moves through consensus.
4. How the application boundary turns local entropy into a shared commitment.
5. What persistence buys you when the node restarts.
6. Why the example omits raw broadcast, backfilling, and retrieval.
7. How to read the source without getting distracted by the UI.

## 4. System Concepts To Explain

- **Secret payload vs. public digest** - the payload stays local, the digest is
  the thing the network agrees on.
- **Private payload vs. public commitment** - the chapter should repeat this
  distinction until it feels obvious.
- **Genesis as the first commitment** - the example hashes a fixed genesis
  message to seed consensus.
- **Application boundary** - consensus asks for digests and verification; the
  application decides how those digests are produced.
- **Persistence** - consensus state lives in `storage-dir`, so restart resumes
  the same story instead of inventing a new one.
- **Observability without protocol coupling** - the GUI and reporter help
  humans understand the run without changing the protocol itself.
- **Omitted broadcast/backfill** - the example leaves raw payload distribution
  as future work so the chapter can stay focused on the commitment problem.

## 5. Visuals To Build Later

1. **Envelope and Receipt Plate** - show a participant placing a secret note in
   an envelope, hashing it, and handing only the digest to consensus.
2. **Digest Agreement Timeline** - show proposal, notarization, and finalization
   as stages where the same digest becomes a shared fact.
3. **Restart and Recovery Diagram** - show consensus state written to
   `storage-dir`, then resumed after shutdown.
4. **What This Example Does Not Do** - a small editorial plate contrasting
   digest agreement with payload broadcast and backfilling.

## 6. Claims To Verify

- [ ] The proposal payload is 16 random bytes generated by the application.
- [ ] Genesis hashes the fixed message `commonware is neat`.
- [ ] The application only exposes digests to consensus, not the raw secret.
- [ ] `verify` returns `true` because the example treats digest validity as a
      consensus concern.
- [ ] `broadcast` is intentionally empty.
- [ ] Consensus state is persisted under the configured `storage-dir`.
- [ ] The example uses `commonware_consensus::simplex` with `RoundRobin`
      leader election.

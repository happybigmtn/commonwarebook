This chapter covers the full p2p stack: how peers prove who they are, how they learn where other peers live, how encrypted connections come into existence, and how application traffic moves through priority lanes once the link is live.

Backgrounder: What Peer-to-Peer Networking Is Actually Hard About

One machine opens a socket, another accepts it, and bytes flow. That beginner picture is fine for a toy client and server. It leaves out the hard parts that matter once the network is dynamic, adversarial, or long-lived.

Peer-to-peer systems have to solve several problems at once. They must discover peers whose addresses may change. They must authenticate identities that are not the same thing as IP addresses. They must keep urgent control traffic from being trapped behind bulky routine traffic. And they must survive a world where some peers are stale, slow, unreachable, or malicious.

Identity Is Not Location

The first serious mental shift is to separate where traffic is sent from who is on the other end. An address is a route hint. A public key is a durable name. If you dial 203.0.113.8:9000, you still do not know whether the remote side is the peer you intended to reach. That is what the handshake is for.

This distinction pays for itself immediately. DNS may resolve to several addresses. A peer may move networks. NAT may make the address you dial different from the address packets appear to come from. A stale record may keep circulating after the machine it described has already moved. Once you accept that identity and address are different facts, much of serious p2p design starts making sense.

Discovery Is a Knowledge Problem

Discovery is not just bootstrap. It is distributed knowledge management. A node needs to know which identities belong in the network, which of them appear reachable right now, how fresh that reachability information is, and which facts are safe to re-share to others. If those different kinds of knowledge are collapsed into one generic idea of a “known peer,” the overlay cannot represent uncertainty honestly.

Handshakes and Channels Solve Different Problems

A handshake does more than start encryption. It proves that the live transport session is bound to the expected identity and namespace. After that, channels, priorities, and backpressure solve a different problem: several classes of messages share one connection, but they should not all have the same urgency. Good p2p systems therefore separate identity proof from traffic management.

NAT, Asymmetry, and Failure Models

Real networks are messy. Some peers can be dialed but not redialed. Some advertise one address yet emit packets from another. Some information is stale but not malicious. Some peers are simply overloaded. A robust overlay has to name those cases separately instead of flattening them into “the connection failed.”

Background Takeaway

Good p2p networking is not socket programming plus retries. It is identity management, knowledge propagation, and contention control under moving addresses and imperfect trust.

What Problem Does This Module Solve?

At first glance, peer-to-peer networking sounds simple: open a socket, connect to a remote address, and send bytes. But that picture leaves out almost everything an adversarial network cares about. How do you know the thing on the other end is really the peer you meant to reach? How do you avoid one slow channel clogging all the other traffic? How do you discover addresses when the network membership changes?

commonware-p2p is Commonware's answer to those questions. It provides authenticated, encrypted, multiplexed communication between peers identified by cryptographic public keys. It operates in two distinct modes:

Discovery mode (discovery::Network) — use this when you know who belongs in the network but not yet where those peers are. Nodes bootstrap from a seed set, gossip signed peer records, and gradually learn the current dialable addresses.

Lookup mode (lookup::Network) — use this when the application already knows both identity and address. The application supplies (PublicKey, Address) pairs directly, so the network can skip the gossip loop entirely.

Both modes share the same actor topology, encrypted stream handshake, priority relay, and per-channel rate limiting. The only real difference is where addresses come from.

The crate composes with the rest of Commonware as follows:

  • Cryptography (commonware-cryptography) — provides the Signer and PublicKey traits used for peer authentication and signed peer attestations.
  • Codec (commonware-codec) — provides Encode/Decode for all wire messages (Ingress, Address, Payload).
  • Runtime (commonware-runtime) — provides the abstract ContextCell that drives all async actors and I/O, enabling deterministic testing via deterministic::Runner.
  • Stream (commonware-stream) — provides the encrypted stream protocol (commonware_stream::encrypted) that handles the actual connection establishment and framing.

Mental Model

Think of commonware-p2p as an encrypted postal service with named mail slots.

  • A peer's public key is its identity.
  • An address is only a route to that identity.
  • A connection is an authenticated encrypted tunnel between two identities.
  • A channel is a named lane inside that tunnel, so unrelated traffic does not all compete in one queue.
  • A priority flag decides whether a message enters the fast lane or the ordinary lane.
Invariant

Identity is never assumed from an address. If a packet arrives from 1.2.3.4:9000, that tells you where it came from, not who sent it. The handshake is what ties the connection to a cryptographic identity.

One subtle detail is worth keeping in mind from the beginning. The Info structure signs (ingress, timestamp), not every field in the struct. So the correct question to ask is not "is this record signed?" but "what exactly does this signature prove?" Good protocol reading often comes down to details like that.

Core Abstractions and Important Types

3.1 — Addressing — Ingress and Address

p2p/src/types.rs

Start with the addressing types, because they explain a design choice that appears everywhere else: Commonware separates what we dial from what we expect packets to arrive from.

Ingress is what we dial to reach a peer:

pub enum Ingress {
    Socket(SocketAddr),                          // Direct IP address
    Dns { host: Hostname, port: u16 },           // DNS name (resolved at dial time)
}

Address is the full address specification:

pub enum Address {
    Symmetric(SocketAddr),                                     // Same address for dial and filter
    Asymmetric { ingress: Ingress, egress: SocketAddr },      // Dial one thing, expect another on receive
}

Asymmetric models NAT traversal. A peer may know the public address others should dial, but inbound traffic may still appear from a different egress IP that must be checked. If those ideas were collapsed into one field, the runtime would either reject valid NATed peers or accept spoofed sources too easily.

Source

Private IP policy: Ingress::Socket is validated against allow_private_ips. Ingress::Dns is resolved first, then the resolved IPs are checked. DNS resolution is randomized — a different IP is selected from the resolved set on each dial attempt, which provides some load balancing.

3.2 — The Wire Protocol — Payload

p2p/src/authenticated/discovery/types.rs

Four Payload variants traverse the wire:

Variant Role When Sent
Greeting(Info<C>)Introduces the peerFirst message on every new connection
BitVec(BitVec)Discovery + keepalivePeriodically (default: every 50s)
Peers(Vec<Info<C>>)Gossip responseWhen BitVec reveals unknown peers
Data(Data)Application payloadWhenever the app sends

BitVec is a compact bitvector where bit i in position index means "I believe I can dial peer sorted_peers[i]." The bitvector is lossy: a 1 bit means knowledge; a 0 means either "unknown" or "dialing failed after dial_fail_limit attempts." This prevents stale records from trapping peers.

3.3 — Signed Peer Records — Info and InfoVerifier

Info is a signed attestation by peer P of its own (ingress, timestamp):

pub struct Info<C: PublicKey> {
    pub ingress: Ingress,
    pub timestamp: u64,       // Epoch milliseconds
    pub public_key: C,
    pub signature: C::Signature,
}
Key Detail

The signed payload is narrower than the whole struct. Only (ingress, timestamp) is covered by the signature. The public_key field tells the verifier which key to use, but it is not itself part of the signed message.

InfoVerifier::validate applies three checks on every gossip message:

  1. Not-self: reject if info.public_key == me — prevents self-announcement.
  2. Timestamp within synchrony_bound: reject if info.timestamp is more than synchrony_bound milliseconds in the future — bounds how far into the future a forged timestamp can be.
  3. Valid signature: verify against the gossip namespace — prevents cross-namespace replay. The signature covers only (ingress, timestamp), not the public_key field.
Why This Matters

This is exactly the kind of detail that separates protocol reading from surface reading. A record can be “signed” and still leave some fields outside the signed claim, so the right question is always: what fact does the signature actually prove?

The gossip namespace _COMMONWARE_P2P_TRACKER is distinct from the stream signing namespace _COMMONWARE_P2P_STREAM. This separation ensures that peer records gossiped through discovery cannot be replayed as stream messages.

3.4 — Priority Relay — Relay<T>

p2p/src/authenticated/relay.rs

Every peer-to-peer channel is backed by a Relay:

pub struct Relay<T> {
    low:  mpsc::Sender<T>,   // Bounded — fills up if app is slow
    high: mpsc::Sender<T>,   // Unbounded — priority messages skip queue
}

send(message, priority=true) routes to high. send(message, priority=false) routes to low. When the low lane is full, try_send returns an error immediately — the relay uses non-blocking semantics, not blocking. The router handles this error by dropping the message and incrementing a messages_dropped_total metric. Priority senders always succeed if the channel is live (their high lane is unbounded).

Mechanism

The important detail is that the low lane is bounded and non-blocking. When it is full, the relay returns an error immediately and the router decides how to absorb the backpressure. That keeps slow low-priority traffic from turning into a hidden stall.

3.5 — Mailbox Primitives — Mailbox and UnboundedMailbox

p2p/src/authenticated/mailbox.rs

The interesting question here is not what sender type sits inside the wrapper. The interesting question is what kind of time pressure this actor is supposed to feel.

At the router and spawner boundary, waiting is informative. If application traffic or connection work arrives faster than the actor can absorb it, the system wants to feel that pressure immediately. A bounded Mailbox<T> turns the queue into a visible limit, which is exactly right for data-plane work where backpressure is part of the contract.

The tracker is different. It is part of the control plane that keeps the rest of the network alive. If that bookkeeping loop blocked on its own mailbox, the network could stall on the very metadata it needs in order to recover. That is why UnboundedMailbox<T> exists. It is not merely “the same mailbox but bigger.” It is the spelling for a different systems claim: control-plane bookkeeping should not stop because the bookkeeping channel itself became the bottleneck.

3.6 — The Shared Trait Surface — Manager, AddressableManager, Provider

p2p/src/authenticated/mod.rs defines the traits that both discovery and lookup implement:

// For discovery: addresses unknown, only Set<PublicKey>
pub trait Manager: Provider {
    fn track(&mut self, id: u64, peers: Set<PublicKey>) -> impl Future<Output = ()>;
}

// For lookup: addresses known, Map<PublicKey, Address>
pub trait AddressableManager: Provider {
    fn track(&mut self, id: u64, peers: Map<PublicKey, Address>);
    fn overwrite(&mut self, peers: Map<PublicKey, Address>); // In-place address updates
}

The Oracle returned by Network::new implements Manager in discovery mode and AddressableManager in lookup mode. The application uses the oracle to register peer sets and (in lookup mode) push address updates.

Execution Flow and Lifecycle

The chapter content covers four flows: startup sequence, outbound connection, inbound connection, and application send. The actor topology plate above (Figure 1) shows the five actors and their mailboxes in a horizontal flow diagram.

The outbound dial path flows: dialer.tick() → tracker.dialable() → reservation → encrypted handshake → spawner → tracker.connect() → router.ready().

The inbound accept path flows: listener.accept() → handshake → tracker.listen() → spawner → tracker.connect() → router.ready().

The application send path: sender.send() → router mailbox → relay.send() → peer actor → encrypted stream. If low-priority and the relay's low lane is full, the router drops the message and increments a metric. High-priority messages always go through the unbounded high lane.

Source

The full startup sequence, connection flows, and application send path are documented in p2p/src/authenticated/discovery/network.rs and p2p/src/authenticated/discovery/actors/router/actor.rs. The Metadata enum in the tracker carries who initiated each connection and (for listener) the observed egress IP.

Concurrency, Protocol, and Systems Semantics

This is where the earlier pieces meet real network pressure: fan-out, stale knowledge, retries, uneven traffic, and slow peers. Read this section less as "what knobs exist?" and more as "what kinds of bad network behavior is this system trying to turn into something survivable?"

The five actors communicate via message passing with fixed mailbox sizes:

Actor Mailbox Bound/Unbounded Backpressure
trackerUnboundedMailboxUnboundedNever blocks sender
routerMailboxBounded (mailbox_size)App sender blocks at Mailbox when full; relay drops low-priority via try_send
spawnerMailboxBounded (mailbox_size)Blocks listener/dialer when full; connection stalls until slot frees
listenerN/A (event loop)—Rate limits via KeyedRateLimiter
dialerN/A (event loop)—select_loop! on sleep tick

The bounded mailboxes on router and spawner provide backpressure: if the router's event loop falls behind, sender.send() blocks at the mailbox_size threshold rather than growing memory indefinitely.

Discovery gossip uses an epidemic broadcast with compact bitvectors. With N fully-connected honest peers and zero failures, all bits converge to 1 within ⌈log₂(N)⌉ gossip rounds. This bound is theoretically sound for epidemic broadcast with full connectivity; the source does not include a machine-checked proof of this property.

Failure Modes, Cancellation, and Correctness Concerns

Networking code is where vague explanations stop being harmless. A peer may be malicious, merely slow, or just reachable from an address you were not expecting. The point of these failure sections is to separate what the protocol forbids from what it merely mitigates.

What a Byzantine Peer Cannot Do

Given valid Info signatures and a correct InfoVerifier: cannot forge address announcements, cannot announce a future timestamp, cannot announce self as another peer, cannot trigger unbounded memory growth, and cannot block honest peers indefinitely (block_duration is finite).

How to Read the Source Code

Do not read the p2p crate as one giant actor graph all at once. Read it in layers: identity and addressing first, then wire format, then actor ownership, and only then the live event loops. That order keeps the transport design visible.

A recommended reading order for the commonware-p2p source:

  1. p2p/src/types.rs — All addressing semantics (Ingress, Address, DNS resolution, private IP filtering, wire encoding). Every subsequent file assumes familiarity with these types.
  2. p2p/src/authenticated/mod.rs — The trait surface (Manager, AddressableManager, Sender, Receiver, Blocker). Understanding what the Oracle actually implements clarifies the two modes.
  3. p2p/src/authenticated/mailbox.rs — Mailbox (bounded) and UnboundedMailbox wrappers around mpsc::Sender. Read this to understand the backpressure primitive before tracing it through actors.
  4. p2p/src/authenticated/discovery/types.rs — Payload, Info, InfoVerifier, BitVec. This is where the security model is enforced.
  5. p2p/src/authenticated/discovery/network.rs — The Network::new constructor shows how all five actors are wired together.
  6. p2p/src/authenticated/discovery/actors/tracker/actor.rs — The tracker owns Directory, Records, Reservations, and processes all tracker message types.
  7. p2p/src/authenticated/discovery/actors/router/actor.rs — The router holds BTreeMap<P, Relay<EncodedData>> and processes Ready, Release, Content, and SubscribePeers messages.
  8. p2p/src/authenticated/discovery/actors/spawner/actor.rs — The spawner is a supervisor that bridges connection establishment to peer actors. Calls tracker.connect() and router.ready() on handoff.
  9. p2p/src/authenticated/discovery/actors/listener.rs — Inbound connection handler. Applies IP/subnet rate limits, private IP filtering, and concurrent handshake limits before handing off to handshake().
  10. p2p/src/authenticated/discovery/actors/dialer.rs — Timed outbound connection loop. Resolves DNS, picks a random resolved address, and calls encrypted::dial().
  11. p2p/src/authenticated/relay.rs — try_send semantics — priority senders never block, low-priority senders return an error immediately when the low lane is full.
  12. p2p/src/simulated/mod.rs — Read last to understand the test infrastructure.
Key pattern

Reservation and Releaser: Reservation holds a Releaser which sends a Message::Release on drop. Any early return in the dialer or listener automatically releases the tracker reservation, preventing leaks.

Glossary and Further Reading

Bitvector (BitVec)
A compact representation of which peers a node knows how to dial, encoded as a bitmap over a sorted peer list at a given epoch index.
Bootstrapper
A (PublicKey, Ingress) pair used to seed discovery. Peers attempt to dial bootstrappers first, then learn about the rest of the network via gossip.
Egress IP
The source IP address observed on an inbound connection. Used in Address::Asymmetric for NAT traversal and spoofing prevention.
Info
A signed attestation (ingress, timestamp) used in gossip. Prevents address forgery. Note: public_key is not covered by the signature.
Oracle
The handle returned by Network::new. Implements Manager (discovery) or AddressableManager (lookup).
Payload
The four-variant enum (Data, Greeting, BitVec, Peers) that forms the entire wire protocol.
Priority relay
A two-lane mpsc facade where priority=true messages go through an unbounded high lane and priority=false messages go through a bounded low lane. When low lane is full, try_send returns an error immediately.
Reservation
A tracker-issued permit to dial or accept a connection. Carried from dialer/listener to spawner to tracker to release.
Synchrony bound
A Duration that bounds how far into the future a gossip timestamp can be. Protects against clock skew and replay.

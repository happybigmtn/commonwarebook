This chapter now opens with a backgrounder on runtimes as a problem class: what they schedule, why async behavior becomes slippery, and why deterministic simulation changes how distributed systems get tested.

Backgrounder: What a Runtime Is Really Doing

Most people first meet a runtime as a piece of ambient magic. You write async, you write await, and "the runtime" somehow makes the rest happen. That beginner picture is useful, but it hides the decisions that matter most in systems work.

A runtime is not your business logic. A runtime is the machinery that decides when tasks get polled, when timers wake back up, which outside-world capabilities a task is allowed to use, and what happens when work is cancelled or interrupted halfway through a side effect. In distributed software, those choices are not background detail. They shape correctness.

The naive sequential model says code runs in order, one function blocks until it returns, and time passes in the background. That picture breaks once a node has to accept inbound traffic, retry dials, flush storage, emit heartbeats, and race those tasks against deadlines. Now the system needs a policy for choosing what runs next. That policy is one of the runtime's core jobs.

Vocabulary That Helps

A few terms carry most of the conceptual load. A task is one unit of async work. A future is a computation that may not be ready yet. An executor polls ready futures. A scheduler is the policy for ordering them. A reactor notices external events, such as sockets becoming readable or timers expiring. Cancellation means a task stops before its normal return point. Backpressure means the system forces a producer to slow down instead of buffering forever.

Those words matter because they separate distinct responsibilities. If you blur them together, it becomes hard to see whether a bug came from the protocol logic, from the scheduling policy, from timer behavior, or from an I/O boundary.

Why Async Bugs Feel Random

They usually are not random. They are hidden scheduling decisions. One task wakes before another publishes state. A timeout matures just before a sibling finishes. A write reaches storage but the metadata update does not. The application experiences that as a flaky failure, but underneath it is the runtime choosing one interleaving out of many possible interleavings.

This is the hard lesson behind many distributed failures: the outside world is part of the trace. The clock, the network, storage, and the executor are all co-authoring the execution. If they stay implicit, tests borrow too much entropy from the host operating system to be replayable.

  • Time affects deadlines, retries, and heartbeats.
  • Network affects arrival order, loss, and wakeups.
  • Storage affects what survives a crash boundary.
  • Scheduling affects which task publishes or observes state first.

Why Determinism Matters

Running the real system many times is useful, but it does not solve replay. If one failure depends on a particular task order and one particular fault point, "run it again" is not much of a debugging strategy. Determinism changes that. It does not mean there is only one schedule. It means the schedule comes from an explicit, reproducible source instead of ambient kernel behavior.

A seeded executor can still explore many interleavings. The difference is that seed 42 can be replayed. The same idea applies to logical time and fault-injected storage. Simulated time is not fake time; it is time whose advancement is defined by the executor. In-memory storage is not pretending to be a real disk platter; it is exposing the semantics the protocol actually cares about, such as partial writes, ordered visibility, and recovery boundaries.

Mental Shift

Deterministic testing is not about making the world simpler than production. It is about making the world explicit enough to replay.

Why Lifetime Structure Matters Too

A runtime is not only an I/O abstraction. It is also a lifetime discipline. Spawned work needs ownership. If a parent task times out or shuts down, its children should not quietly keep mutating storage or waiting on timers after the rest of the system believes the subtree is gone. That is the intuition behind structured concurrency: task trees should reflect responsibility.

Capability boundaries matter for the same reason. If protocol code can reach out to global clocks, arbitrary file paths, and detached threads whenever it wants, the runtime cannot control or replay much of anything. If the task instead receives a context with explicit clock, storage, spawn, and network capabilities, the same logic can run in a simulator or against production backends without rewriting the protocol body.

The Bridge Into Commonware

That is the right frame for commonware-runtime. The crate is not trying to be a generic async fashion statement. It is solving a more pointed problem: Commonware protocols need to run against a real production runtime and also against a deterministic world where scheduling, timers, storage faults, and network behavior can be replayed on demand.

The rest of this page narrows from that broad systems background to the specific Commonware mechanism: the abstract executor boundary, the deterministic runner, the supervision tree, the storage model, and the production Tokio backend. Once the general problem is clear, those pieces stop looking like plumbing and start looking like one coherent answer to a hard question: how do you keep concurrent protocol code honest when the environment is allowed to misbehave?

The Problem the Runtime Solves

Suppose a test fails once, late at night, after thousands of successful runs. The next morning it passes. Then it passes again. The bug is real, but the conditions that trigger it are slippery: one task woke slightly earlier, another task lost a race for the CPU, and a timeout fired in the one interleaving you were not watching.

That is the problem commonware-runtime is built to solve. Most flaky distributed-system tests are not mysterious. They are just tests that borrow too many decisions from the outside world: the OS scheduler picks who runs first, the clock decides when timeouts expire, the network decides when bytes arrive, and storage decides when writes become visible. All of that is external entropy.

commonware-runtime takes those decisions back by providing an abstract executor interface. The application says what work it wants to do; the runtime decides how that work is scheduled. The same async logic then runs unchanged in two environments:

  • Deterministic simulator (deterministic::Runner): a seeded StdRng drives task selection. Network, storage, and clock are all in-memory traits with pluggable implementations. No syscalls, no kernel involvement. Every flake is reproducible by shipping the seed.
  • Production runtime (tokio::Runner): multi-threaded Tokio scheduler with optional io_uring for fast I/O, OpenTelemetry tracing, and Prometheus metrics.

Key insight

Deterministic simulation is not slow simulation. In the deterministic runner, "network I/O" is often just bytes moving between memory buffers, and "sleeping" is often just an alarm inserted into a heap. A protocol that needs 10 seconds on a real network may simulate in milliseconds.

The crate also gives the rest of the library a disciplined I/O boundary. Every surface that touches the outside world—network, storage, clock, metrics—is a trait. That makes fault injection ordinary instead of heroic: drop 30% of messages, fail every tenth write, corrupt a blob, or freeze time, all without building a fake application around the real one.

Mental Model of the Runtime

The cleanest mental model is a small theater. Tasks are the actors. The executor is the stage manager. The ready queue is the list of actors who can continue right now. The sleeper queue is the list of actors waiting for a future time. The context is the toolkit each actor carries on stage.

  • Tasks: async futures that make progress only when they are polled. When a task hits .await, it is really saying: "I cannot continue now; poll me again when this thing becomes ready."
  • The Executor: the central loop that decides which task to poll next. It owns the ready queue, the sleeper heap, and the logical clock.
  • The Context: the handle handed to every task. It provides the task's capabilities: spawn children, sleep, dial peers, open blobs, and register metrics.
  • The Supervisor Tree: the ownership graph for task lifetimes. If a parent is aborted, its descendants are aborted with it. Siblings are not.

Now make it concrete. Suppose tasks A and B are ready, while task C is sleeping until 5ms. The executor drains the ready queue, maybe shuffles [A, B], polls each once, advances logical time, and then asks whether any sleeper deadline has passed. If C's deadline has arrived, C is moved back into the ready queue for the next cycle. 

Runner.start(|context| async move {
    // context: the root of the supervisor tree
    let child = context.clone().spawn(|child_ctx| async move {
        // child_ctx: a child in the supervisor tree
        child_ctx.sleep(Duration::from_secs(1)).await;
        "done"
    });
    // If the parent (root) is aborted here, `child` is also aborted.
    // If `child` panics, only `child` and its descendants are aborted.
});

In the deterministic runtime, the executor loop is: 

loop {
    // 1. Drain all ready tasks
    ready_tasks = tasks.drain();

    // 2. Shuffle with seeded RNG (if > 1 task)
    if ready_tasks.len() > 1 {
        ready_tasks.shuffle(&mut rng);
    }

    // 3. Poll each task once
    for task in ready_tasks {
        poll(task);
        if task.is_complete() { remove(task); }
        if root_is_complete() { return result; }
    }

    // 4. Advance clock by cycle duration
    advance_time(CYCLE);

    // 5. Wake all sleepers whose deadline has passed
    wake_ready_sleepers(now);

    // 6. Assert liveness (stall detection)
}

Invariant

The cycle duration (default 1ms) prevents a CPU-starved task from monopolizing the executor. After every full pass through the ready queue, the clock advances by one cycle. A task that never yields still gets only one poll per cycle.

Core Abstractions and Important Types

Runner — runtime/src/lib.rs:140

The entry point trait. start(context) -> Fut::Output takes a closure that receives a Context and returns a future. The executor drives that future to completion. Both deterministic::Runner and tokio::Runner implement this trait—the application code is entirely agnostic to which is running. 

pub trait Runner {
    type Context;
    fn start<F, Fut>(self, f: F) -> Fut::Output
    where
        F: FnOnce(Self::Context) -> Fut,
        Fut: Future;
}

Context — runtime/src/deterministic.rs:887

The handle given to every task. Combines Spawner + Clock + Network + Storage + Metrics. When you clone() a context or call with_label(), you get a child context attached to a new node in the supervisor tree.

Source

runtime/src/deterministic.rs lines 887–940 · The struct holds a hierarchical name, a scope (Arc<ScopeGuard>) for scoped metrics deregistration, a Weak<Executor> to avoid cycles, an Arc<Network>, an Arc<Storage>, an Arc<Tree> for the supervisor node, and an Execution mode flag.

Clock — runtime/src/lib.rs:466

Abstraction over time. In the deterministic runtime, Clock::current() reads from a Mutex<SystemTime> that only advances when the executor calls advance_time(). A task that never calls sleep() or timeout() never advances the clock—time is entirely cooperative in simulation.

Failure mode

There is no public context.advance() method. Time always advances through the executor's internal advance_time() call or via fired alarms. A task that never awaits will freeze the deterministic clock indefinitely.

Storage and Blob

The Blob interface uses explicit offsets rather than streaming I/O. read_at(offset, len) and write_at(offset, bufs) are atomic at an offset—enabling sparse writes and append-only friendliness.

Header — runtime/src/storage/mod.rs:62

Every blob on disk begins with an 8-byte header:

Bytes Field Value
0-3 MAGIC b"CWIC" (Commonware Is CWIC)
4-5 Runtime Version u16, big-endian · 0 = current
6-7 Blob Version u16, big-endian · varies

Header::missing(raw_len) returns true when raw_len < 8, triggering recovery (truncate and re-initialize) on open. This is the mechanism for crash recovery: after an unclean shutdown, the blob is reopened, the header validation fails, and the storage implementation re-initializes to a clean state.

Execution Flow and Lifecycle

Spawning a Task

When context.spawn(f) is called:

  1. A new Tree::child() node is created as a descendant of the current node.
  2. Tasks::register_work() allocates a new task ID, wraps the future in Pin<Box<dyn Future>>, and inserts it into the running BTreeMap.
  3. The task ID is pushed onto the ready queue.
  4. Tree::register(aborter) records the abort handle on the tree node.

The child future starts running immediately (not lazily)—even if the caller never awaits the returned Handle.

Source reference

The executor loop is at runtime/src/deterministic.rs:534-662. The shuffle step is the source of non-determinism: given the same seed and same initial task set, the RNG produces the same task order. Different seeds produce different orderings—enabling exhaustive exploration of the scheduling state space.

Waker Registration

When a future yields (Poll::Pending), it must register a waker via cx.waker().clone(). In the deterministic runtime, the waker is a TaskWaker (runtime/src/deterministic.rs:752): 

struct TaskWaker {
    id: u128,
    tasks: Weak<Tasks>,
}

impl ArcWake for TaskWaker {
    fn wake_by_ref(arc_self: &Arc<Self>) {
        if let Some(tasks) = arc_self.tasks.upgrade() {
            tasks.queue(arc_self.id);  // re-enqueue the task
        }
    }
}

wake_by_ref() upgrades the weak reference to the task queue and pushes the task ID onto the ready queue. The task will be polled again in the next executor cycle.

Clock Advancement

The deterministic clock starts at UNIX_EPOCH (configurable via Config::with_start_time) and advances only when:

  • The executor's cycle completes (every full pass through the ready queue).
  • A sleep_until deadline is reached and wake_ready_sleepers fires.
  • In external mode, the pacer constrains how much logical time a future can consume per cycle.

Concurrency, Scheduling, and Time Semantics

Up to this point, the runtime can sound like a collection of traits and helper structures. This is the section where those pieces become a theory of behavior. The right question to keep asking is not "what helper does this call?" but "what guarantees does this scheduling model buy us, and what does it refuse to pretend to buy us?"

Task Scheduling

The deterministic executor uses cooperative multitasking with a seeded shuffle. Every cycle:

  1. All tasks in the ready queue are collected.
  2. The RNG shuffles the order.
  3. Each task is polled once.
  4. The clock advances by cycle (1ms default).
  5. Sleepers whose deadlines have passed are woken.

This means scheduling is non-preemptive within a cycle: a task that never awaits gets one poll per cycle, not continuous CPU. The 1ms cycle prevents starvation for I/O-bound tasks that properly await.

The useful comparison is with Tokio. Tokio is trying to keep real hardware busy. The deterministic runner is trying to keep causality legible. Tokio uses real threads and work stealing because it cares about throughput on a live machine. The deterministic runner polls in a seeded order because it cares about whether you can replay the exact same interleaving tomorrow.

Structured Concurrency

The supervisor tree enforces that child tasks are cancelled when their parent is aborted. This is structured concurrency without the mailbox: it is purely about cancellation propagation, not message passing.

Put more plainly: if you created the work, you own its lifetime. The tree is the runtime's way of enforcing that promise instead of leaving it up to human discipline.

The tree is a DAG of Arc<Tree> nodes with Option<Arc<Tree>> parent pointers and Vec<Weak<Tree>> child lists. When a node aborts, it takes its task handle and drains its children vector, recursively aborting each child via Weak::upgrade(). Siblings are untouched.

Invariant

The tree uses weak references throughout. If a child task is dropped without being aborted (normal completion), the weak reference becomes invalid but the parent's children list is cleaned up lazily on the next child() call via retain(|weak| weak.strong_count() > 0).

Metric Scope Lifecycle

Metrics registered via context.register("name", "help", metric) are prefixed with the context's label. The ScopeGuard (runtime/src/utils/mod.rs:437) enables automatic deregistration when a scope ends. 

pub(crate) struct ScopeGuard {
    scope_id: u64,
    cleanup: Option<Box<dyn FnOnce(u64) + Send + Sync>>,
}

impl Drop for ScopeGuard {
    fn drop(&mut self) {
        if let Some(cleanup) = self.cleanup.take() {
            cleanup(self.scope_id);  // deregisters metrics for this scope
        }
    }
}

Failure Modes, Cancellation, and Correctness Concerns

This section is the shadow of the design. Every abstraction looks neat on the happy path. The real question is what happens when the world is late, malicious, partial, duplicated, or half-written. The point of the runtime is not just to run work; it is to make those ugly cases explicit and testable.

Stall Detection

The deterministic executor panics if the ready queue is empty and no sleepers exist: 

fn assert_liveness(&self) {
    if cfg!(feature = "external") || self.tasks.ready() != 0 {
        return;
    }
    panic!("runtime stalled");
}

This catches cases where all tasks are waiting on external events that never fire. In external mode, the executor instead sleeps for a cycle and re-checks.

Conceptually, a stall is the runtime saying: “I can see the whole world, and nothing in it can ever make progress again.” That is why the check is so valuable in tests. It turns a silent hang into an explicit failure.

Correctness concern

If a future is polling a blocking syscall (e.g., tokio::fs::File::read) when aborted, the OS may continue the syscall until it returns—at which point the result is discarded because the task is already marked aborted. This is standard Tokio behavior; the abort is cooperative at the future level.

Storage Fault Injection

FaultyStorage wraps an inner storage and injects failures deterministically. The failure rate is a probability (0.0–1.0) per operation type, drawn from the shared RNG. 

let faults = FaultConfig::default()
    .write(0.1)           // 10% of writes fail
    .partial_write(0.5);   // Of failed writes, 50% are partial
let storage = FaultyStorage::new(inner, rng, Arc::new(RwLock::new(faults)));

Partial writes write some bytes before failing—testing that recovery handles truncated data. This is the key advantage over ad-hoc mocking: you can test the recovery path, not just the happy path.

Blob Version Mismatch

When a blob's version is not in the requested versions range, open_versioned returns Error::BlobVersionMismatch { expected, found }. This enables forward-compatibility testing: a storage reader can specify a version range it understands, and if the blob was written by a newer version outside that range, the open fails explicitly rather than silently ignoring the version field.

Failure mode

If the magic bytes are invalid (not b"CWIC"), Header::from() returns HeaderError::InvalidMagic, which is wrapped into Error::BlobCorrupt. The caller must handle this explicitly. Note the distinction: truncation triggers automatic recovery; invalid magic requires explicit error handling and is not recoverable.

Tokio Runtime io_uring

When the iouring-storage and/or iouring-network features are enabled, the Tokio runtime uses io_uring for async I/O with kernel-side buffer recycling. This is significantly faster than epoll + read/write for storage-heavy workloads because it avoids a memory copy between kernel and userspace.

How to Read the Source Code

Do not read the runtime source as if it were a dictionary. Read it as if you were tracing an argument. Start from the execution model, then look at the capability traits, and only then look at the helper structures that make those capabilities testable.

The trait hierarchy is in runtime/src/lib.rs, grouped by stability level. The Error enum at line 82 enumerates every way a runtime operation can fail.

Recommended reading order

  1. Entry point — runtime/src/lib.rs: trait hierarchy, Error enum, stability annotations
  2. Deterministic executor — runtime/src/deterministic.rs: Config (line 205), Auditor (line 161), TaskWaker (line 752), Tasks (line 770), Alarm (line 1382), Context (line 887), start_and_recover() (line 534)
  3. Mock channel — runtime/src/mocks.rs: Channel with Sink/Stream for deterministic network
  4. Storage stack — runtime/src/storage/: layered as MeteredStorage<AuditedStorage<FaultyStorage<MemStorage>>>
  5. Buffer types — runtime/src/iobuf/mod.rs: IoBuf, IoBufMut, IoBufs, BufferPool
  6. Supervision — runtime/src/utils/supervision.rs: weak-reference abort tree, retain-based lazy cleanup

Source note

The executor loop (deterministic.rs:556-662) drains the ready queue, shuffles with the RNG, polls each task, advances the clock, and wakes sleepers. The Checkpoint capture at the end enables stateful replay: halt mid-execution, corrupt a blob, reopen, and verify the recovery path.

Glossary

Term Definition
Deterministic runtime A runtime that produces identical task execution order when run with the same seed. Uses a seeded StdRng to shuffle the ready queue each cycle.
Cooperative multitasking Tasks yield control by returning Poll::Pending and registering a waker. The executor re-queues them when the waker fires. No preemption within a cycle.
Supervisor tree A DAG of weak-reference nodes tracking the task hierarchy. Parent abort cascades to all descendants.
Alarm A SystemTime + Waker pair stored in a min-heap. Fires when the deterministic clock passes the deadline.
CWIC header 8-byte blob header: 4-byte magic b"CWIC", 2-byte runtime version, 2-byte blob version. Present at the start of every blob.
Fault injection Deliberately introducing failures (crash, corruption, drop) to test recovery paths.
Checkpoint Captured executor state (time, RNG, storage) used to resume simulation from a specific point.
Buffer pool Pool of fixed-size aligned allocations returned to the pool on drop via PooledBuf's Drop impl.
OpenMetrics Prometheus text format with type-specific suffixes (_total for counters, _count/_sum/_bucket for histograms).
io_uring Linux kernel async I/O interface with zero-copy buffer passing between kernel and userspace.
Structured concurrency Enforcing that child tasks are cancelled when their parent exits, via the supervisor tree.
